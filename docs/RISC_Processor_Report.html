1. Project Overview & Objectives

This project implements a fully functional 4-stage pipelined RISC processor using Verilog HDL, modelled on the classic RISC-V RV32I architecture. The design merges the traditional Memory Access and Write Back stages into a combined MEM/WB stage, reducing hardware complexity while retaining the performance benefits of pipelining.

⚙️
4-Stage Pipeline
IF → ID → EX → MEM/WB with full interlocking and forwarding logic.
🔁
Data Forwarding
EX–EX and MEM–EX forwarding resolves most data hazards with zero penalty cycles.
🛑
Hazard Detection
Load-use stall detection inserts NOP bubbles automatically; branch flushes IF/ID.
📦
17 Instructions
ADD, SUB, ADDI, AND, OR, XOR, SLT, SLL, SRL, SRA, LW, SW, BEQ, BNE, JAL, LUI, AUIPC.
🗂️
32 Registers
32×32-bit general-purpose registers; R0 hardwired to zero per RISC-V spec.
🧪
Verified in Vivado
Self-checking testbenches covering forwarding, stalls, branches, and arithmetic.
🎯 Design Philosophy: Modelled after the classic 5-stage RISC pipeline, simplified to 4 stages by merging MEM and WB. This reduces hardware complexity while retaining pipelining throughput benefits. Implemented entirely at RTL level in Verilog HDL.

2. Pipeline Architecture

The processor is organised as four sequential pipeline stages, each separated by pipeline registers that carry state from one clock edge to the next.

🔵 IF — Instruction Fetch

Reads instruction from ROM at the current PC. Computes PC+4. On branch taken, PC is redirected and IF/ID is flushed. PC is frozen (held) during a load-use stall.

// PC next logic pc_next = branch_taken ? branch_target : pc + 4;

🟢 ID — Instruction Decode

Simultaneously decodes the opcode (control unit), reads two source registers, sign-extends the immediate. The hazard detection unit checks for load-use conflicts and issues stalls.

// Hazard detection condition if (id_ex_mem_read && (id_ex_rd == if_id_rs1 || id_ex_rd == if_id_rs2)) → stall + NOP bubble

🟣 EX — Execute

Forwarding unit selects correct ALU operands (EX/MEM or MEM/WB registers). ALU control decodes funct3/funct7 to a 4-bit operation code. Branch target computed as PC + imm.

// Forwarding MUX A forward_a = 2'b10 → EX/MEM result forward_a = 2'b01 → MEM/WB result forward_a = 2'b00 → register file

🟠 MEM/WB — Memory + Write Back

LW reads from data memory; SW writes. Branch decision made here — if taken, flushes IF/ID. Write-back MUX selects between ALU result and memory read data to write to the register file.

// Write-back MUX wb_data = mem_to_reg ? mem_read_data : alu_result;

3. Module Hierarchy & RTL Design

The processor is implemented as a hierarchical RTL design with 14 sub-modules instantiated in the top-level risc_processor.v.

ALU Module (alu.v)

Forwarding Unit (forwarding_unit.v)

module forwarding_unit ( input [4:0] ex_rs1, ex_rs2, input ex_mem_reg_write, input [4:0] ex_mem_rd, input mem_wb_reg_write, input [4:0] mem_wb_rd, output reg [1:0] forward_a, forward_b ); // Priority: EX hazard > MEM hazard always @(*) begin // Forward A (RS1) if (ex_mem_reg_write && ex_mem_rd != 5'b0 && ex_mem_rd == ex_rs1) forward_a = 2'b10; // EX-EX else if (mem_wb_reg_write && mem_wb_rd != 5'b0 && mem_wb_rd == ex_rs1) forward_a = 2'b01; // MEM-EX else forward_a = 2'b00; // reg file // Forward B (RS2) — same logic ... end endmodule

Hazard Detection (hazard_detection.v)

module hazard_detection ( input id_ex_mem_read, input [4:0] id_ex_rd, input [4:0] if_id_rs1, if_id_rs2, output reg pc_write, output reg if_id_write, output reg ctrl_mux_sel ); always @(*) begin if (id_ex_mem_read && (id_ex_rd == if_id_rs1 || id_ex_rd == if_id_rs2)) begin // Load-use hazard detected pc_write = 1'b0; // freeze PC if_id_write = 1'b0; // freeze IF/ID ctrl_mux_sel = 1'b1; // NOP bubble end else begin pc_write = 1'b1; if_id_write = 1'b1; ctrl_mux_sel = 1'b0; end end endmodule

Immediate Generator (imm_gen.v)

module imm_gen ( input [31:0] instruction, output reg [31:0] imm_out ); wire [6:0] opcode = instruction[6:0]; always @(*) begin case (opcode) 7'b0010011, // I-type 7'b0000011: // LW imm_out = {{20{instr[31]}}, instr[31:20]}; 7'b0100011: // S-type imm_out = {{20{instr[31]}}, instr[31:25],instr[11:7]}; 7'b1100011: // B-type imm_out = {{20{instr[31]}}, instr[7],instr[30:25], instr[11:8],1'b0}; 7'b0110111: // U-type LUI imm_out = {instr[31:12], 12'b0}; 7'b1101111: // J-type JAL imm_out = {{12{instr[31]}}, instr[19:12],instr[20], instr[30:21],1'b0}; endcase end endmodule

11. Known Limitations & Future Work

⚠️ Current Limitations

1. No Interrupt/Exception Support
No traps, CSR registers, or privilege modes. ECALL/EBREAK are not handled.
2. Single-Cycle Memory
Both instruction and data memories are synchronous ROM/RAM with no latency. Real DRAM integration would require stall logic.
3. Branch Resolved in MEM Stage
A 1-cycle flush penalty on every taken branch. Earlier (ID-stage) branch resolution would eliminate this.
4. No Cache Hierarchy
Direct memory access only. Cache misses would require additional stall logic not present in this design.

🚀 Future Enhancements

1. Branch Prediction
Add a 2-bit saturating counter predictor to eliminate most control hazard penalties.
2. Full RV32I ISA
Implement remaining instructions: SLTI, SLTIU, ANDI, ORI, XORI, SLLI, SRLI, SRAI, AUIPC, JALR.
3. Instruction Cache
Direct-mapped I-cache to reduce instruction fetch latency on larger programs.
4. 5-Stage Pipeline
Split MEM/WB into separate stages to further improve clock frequency.
5. FPGA Peripheral Integration
Add UART, GPIO, and timer peripherals for a minimal SoC implementation on the Basys3 board.
